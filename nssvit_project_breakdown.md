## 🤔 What is Astro? (Beginner Explanation)

Astro is a **web framework** that is "static-first". Here's how to think about it:

- **`.astro` files** = Like HTML templates with superpowers. They can run server-side code (fetch data, query APIs) inside `--- ---` (called the "frontmatter fence"), then output plain HTML.
- **React inside Astro** = Astro lets you use React components inside `.astro` pages. These React components are called **"islands"** — interactive bits dropped into otherwise static pages.
- **`client:load`** directive = When you write `<Navbar client:load />` in Astro, it means "make this React component interactive in the browser". Without it, it would just render as static HTML.

## 🤔 What is Sanity? (Beginner Explanation)

Sanity is a **Headless CMS** (Content Management System). Think of it as a database + admin dashboard for website content.

- The website owner logs into `/admin` to update text, images, events, team members, etc.
- The website fetches that content using **GROQ queries** (Sanity's query language, like SQL for content).
- All images, text, logos, navigation links — everything comes from Sanity, not hardcoded in files.

### Example GROQ Query (from code):
```groq
*[_type == "event" && isUpcoming == true && eventDate > now()] | order(eventDate asc)[0...5]
```
This means: *"Give me upcoming events ordered by date, first 5 results"*

## 🎨 Design System — "Swiss Style"

The creator deliberately followed the **Swiss / International Typographic Style** (clean, minimal, grid-based):

### Colors
| Token | Hex | Usage |
|-------|-----|-------|
| `primary` | `#4338CA` (deep indigo) | Buttons, links, accents |
| `text` | `#1E1B4B` (near-black indigo) | All body text |
| `swiss.white` | `#FFFFFF` | Backgrounds |
| `swiss.gray.*` | 50–900 scale | Borders, subtle fills |
| `status.success/warning/error` | Green/Yellow/Red | Alerts & form feedback |

### Typography
- **Font:** `Inter` (clean grotesque sans-serif — perfect for Swiss style)
- **Monospace:** `JetBrains Mono` (for code blocks)
- Defined named sizes: `display`, `headline`, `title`, `body`, `caption`, `label`

### Key Rule: **NO shadows** — Uses borders instead for separation (Swiss principle)

All 20 content types defined in `sanity/schemas/`:

| Schema | What it controls |
|--------|-----------------|
| `siteConfig` | Logo images, OG image, site title |

|--------|---------------|

| `volunteerGroup` | Groups of volunteers per year |

| `camp` | Special Camp details (their big annual event) |
| `achievement` | Awards and recognitions |
| `galleryImage` | Photo gallery entries linked to events/years |
| `announcement` | Banner notifications on homepage |
| `faq` | FAQ questions and answers |
| `developer` | Website developer credits |
| `contactSubmission` | Contact form submissions (stored in Sanity) |

## 📏 Creator's Rules & Conventions

### 1. File Header Comment (Every Single File)
Every file starts with this ASCII art banner:
```js
/**
 * ╻ NSS-VIT
 * ┃ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 * ┃ Not Me, But You
 * ┃
 * ┃ [filename]
 * ╹ [folder path]/
 */
```

### 2. File Naming Convention
- **Astro components:** `PascalCase.astro` (e.g. `BentoCard.astro`, `PageHeader.astro`)
- **React components:** `PascalCase.tsx` (e.g. `Hero.tsx`, `ContactForm.tsx`)
- **Utilities/schemas:** `camelCase.ts` (e.g. `sanity.ts`, `academicYear.ts`)
- **Pages/scripts:** `kebab-case.astro` / `.mjs` (e.g. `get-involved.astro`, `seed-sanity.mjs`)

### 3. Barrel Exports (`index.ts`)
Each component folder has an `index.ts` that re-exports all components:
```ts
// src/components/home/index.ts — export everything from one place
export { Hero } from './Hero';
export { default as AboutSection } from './AboutSection.astro';
```

### 4. Two Component Types – Hybrid Pattern
- `.astro` files = **Static parts** (server-rendered, no interactivity needed)
- `.tsx` files = **Interactive parts** (needs React for clicks, state, animations)
- This is Astro's "Islands Architecture" — only interactive components load JS

### 5. TypeScript Everywhere
- All types/interfaces defined centrally in `src/lib/sanity.ts`
- All Sanity schema fields have corresponding TypeScript interfaces
- Strong typing throughout — no `any` except for Portable Text (rich text)

### 6. Centralized GROQ Queries
All Sanity queries are organized in `src/lib/sanity.ts` under a `queries` object:
```ts
export const queries = {
  upcomingEvents: `*[_type == "event" && isUpcoming == true...]`,
  teamByYear: (year: string) => `...`,
  // etc.
}
```

### 7. Academic Year as a Core Reference
- Everything (events, team, gallery, achievements) is linked to an `academicYear`
- This enables year-based filtering across the entire site
- There's a `isCurrentYear: boolean` flag on each year

### 8. Swiss Design — No Shadows Rule
Explicitly commented in `tailwind.config.mjs`:
```js
// NO SHADOWS - Swiss Design Principle
// Using borders instead for visual separation
```

### 9. SEO in BaseLayout
All SEO meta tags are in `BaseLayout.astro`, not individual pages:
- OpenGraph tags (for link previews on social media)
- Twitter card tags
- Canonical URLs
- Robots meta

### 10. Navigation is CMS-Controlled
Nav links are fetched from Sanity on every request — admins can reorder/show/hide nav items from the `/admin` panel without touching code.

## 🚀 How to Run Locally

```bash
cd nss-vit
npm install
cp .env.example .env    # Add SANITY_PROJECT_ID, SANITY_DATASET, SANITY_API_TOKEN
npm run dev              # Starts at http://localhost:4321
```

## 💡 Key Things to Know Before Working on This

1. **Most "text" is in Sanity, not in code** — if you can't find where a text string lives in `.astro` files, it's probably fetched from Sanity.
2. **React components need `client:load`** to be interactive — without it they're just HTML.
3. **GROQ is like SQL for Sanity** — `*[_type == "event"]` means "get all events". The `->` dereferences relations.
4. **The `academicYear` reference is everywhere** — it's the primary way data is filtered.
5. **Seeding scripts** (`npm run seed:*`) are for initial data upload — don't run them unless starting fresh.
